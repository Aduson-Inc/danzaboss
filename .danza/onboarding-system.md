# DANZA Onboarding System

Tony Danza must complete this before ANY build begins.

---

## 1. PROJECT OVERVIEW

- What is the name of the app?
- What does the app do (in plain language)?
- Who is the target user?

---

## 2. CORE GOALS

- What are the main features?
- What MUST be included (non-negotiable)?
- What is optional / nice-to-have?

---

## 3. USER FLOW

- How does a user enter the app?
- What are the key actions a user takes?
- What is the desired end result for the user?

---

## 4. AUTHENTICATION

- Does the app require login/signup?
- What type? (email, Google, etc.)
- Any permissions/roles?

---

## 5. DATA & STORAGE

- What data is being stored?
- Where is it stored? (DB, local, API)
- Any sensitive data?

---

## 6. API / EXTERNAL SERVICES

- Are there external APIs?
- What do they provide?
- Any rate limits or constraints?

---

## 7. FRONTEND / UI

- Any design preferences?
- Existing design system?
- Mobile, desktop, or both?

---

## 8. TECH STACK

- Preferred framework?
- Backend language?
- Database?

---

## 9. CONSTRAINTS

- Deadlines?
- Performance requirements?
- Budget limitations?

---

## 10. EDGE CASES

- What could go wrong?
- Any unusual user behavior?

---

## 11. FINAL CHECK

Tony MUST ask:

"Is there anything we haven't covered that could affect how this app works?"

Repeat until user says NO.

---

## 12. VALIDATION

Tony summarizes:

- Project understanding
- Features
- Constraints

User must confirm:

"Yes, proceed."

---

## 13. ONBOARDING COMPLETE

Only AFTER this:

→ Samantha maps system  
→ Tony creates feature list  
→ Execution begins

---

## 14. ONBOARDING EVOLUTION RULE

If ANY missing info is discovered later:

→ Add it to:
.danza/onboarding-misses.md

→ Future onboarding MUST include it
